# Traefik JWT Dynamic Redirect

A custom middleware plugin for **Traefik** that performs a dynamic redirect based on values extracted from a JWT token.

The plugin reads the `Authorization` header (`Bearer token`), decodes the JWT payload, and builds a redirect URL using configurable templates.

This allows you to build URLs dynamically using **JWT claims** and the **token itself**.

---

# Features

* Extract JWT from the `Authorization` header
* Decode JWT payload without external dependencies
* Access JWT claims dynamically
* Build redirect URLs using templates
* Redirect users automatically using values from the token

---

# Use Case

Example JWT claim:

```json
{
  "databaseVersion": "26.1"
}
```

Incoming request header:

```
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...
```

Redirect generated by the plugin:

```
https://myvisual.es/release/26.1/#/login?jwt=eyJhbGciOiJSUzI1NiIs...
```

---

# Configuration

## Static Configuration (Traefik)

Enable the plugin in the Traefik static configuration.

```yaml
experimental:
  plugins:
    jwt-dynamic-redirect:
      moduleName: github.com/achaveztematica/traefik-jwt-dynamic-redirect
      version: v0.1.1
```

---

## Dynamic Middleware Configuration

```yaml
http:
  middlewares:
    jwt-redirect:
      plugin:
        jwt-dynamic-redirect:
          redirectTemplate: "https://myvisual.es/release/{claim.databaseVersion}/#/login?jwt={token}"
```

---

# Template Variables

The plugin supports dynamic variables inside `redirectTemplate`.

| Variable         | Description                                       |
| ---------------- | ------------------------------------------------- |
| `{token}`        | JWT token extracted from the Authorization header |
| `{claim.<name>}` | Any claim present inside the JWT payload          |

Example:

```
{claim.databaseVersion}
```

# How It Works

1. Traefik receives the request.
2. The middleware reads the `Authorization` header.
3. The JWT payload is decoded.
4. Claims are extracted.
5. The redirect template is rendered.
6. The user is redirected with HTTP `302`.

---

# Limitations

* JWT signature validation is not performed (payload decoding only).
* Requires the `Authorization: Bearer` header.
* Only string claims are currently supported in templates.

---

# Security Considerations

Since the plugin decodes the JWT without validating the signature, it should be used only when the token has already been validated by an upstream authentication middleware such as:

* OIDC middleware
* Identity proxy
* API gateway
